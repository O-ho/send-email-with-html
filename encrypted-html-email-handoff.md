# 암호화 HTML 첨부 메일 — 설계·분석 정리 (핸드오프)

새 레포에서 **Node로 암호화한 값을 넣은 HTML을 메일로 보내는 샘플**을 만들 때, 아래 내용을 기준으로 질문·구현하면 된다.  
(기존 `attach.html` VestMail 예시와의 비교, 권장 구조, 역할 분담을 한곳에 모았다.)

---

## 1. 목표

- 투자내역 등 **민감 내용을 HTML**로 만들어 메일 첨부(또는 본문)로 보낸다.
- 수신자는 **본인 확인용 비밀번호**(예: DB의 주민등록번호 앞 6자리)를 입력해야 내용을 볼 수 있다.
- 메일 발송은 **배치**로 돌려도 됨(동기 UX 불필요).

---

## 2. VestMail 스타일 예시 파일(`attach.html`)에서 확인한 동작

- **비밀번호가 HTML 안에 평문으로 박혀 있지 않다.** 실제 본문은 **암호화된 데이터**다.
- UI 문구상 개인 고객은 **「생년월일 6자리」** 안내가 많고, 법인은 사업자번호 뒤 7자리 등 다른 규칙이 있다.  
  - 주민번호 앞 6자리와 생년월일(YYMMDD)은 내국인에게 흔히 같지만, **정책·문구는 별도 검토**가 필요하다.
- 암호문은 **`s[0]`, `s[1]`, `s[2]` 같은 JS 배열에 들어 있는 거대한 Base64 문자열**로 들어 있다. **JSON 필드가 없어도 된다.** 상용 제품은 난독화·자체 포맷을 쓴다.
- 일부 로직은 **`b_p` 같은 Base64를 `eval(atob(b_p))`로 풀어** 실행한다.
- 스크립트에는 **SHA-256**(`window.x.s2` 등), **블록 암호 복호**(AES 계열로 추정), 패딩/CBC류 모드로 보이는 `x.b.D` / `x.b.m` 등이 포함된다.
- **메일 서버가 ZIP/PDF 비밀번호를 거는 방식과 다르다.** 첨부 **HTML+JS가 암호문과 복호 루틴을 함께 포함**하는 **자급자족 파일**에 가깝다.

---

## 3. 권장 아키텍처 (새 구현)

| 단계 | 담당 | 내용 |
|------|------|------|
| 암호화·파일 생성 | **백엔드/배치(Node 등)** | 평문 HTML(또는 템플릿 렌더 결과)을 대칭키로 암호화. 키는 **해당 유저 비밀번호(6자리 등)에서 파생**한 값과 **동일 규칙**으로 클라이언트에서도 재현 가능해야 함. |
| 메일 발송 | 배치 | 생성된 `.html` 첨부 또는 링크. |
| 복호화·표시 | **첨부 HTML 안의 JavaScript** | 사용자 입력 → 동일 KDF/알고리즘으로 키 복원 → 복호 → `iframe` `srcdoc` 또는 `DOMParser`로 표시. |

- **DB의 주민 앞 6자리를 브라우저(웹 앱)로 보내서 거기서 암호화하는 방식은 비권장** — 유출·로그·네트워크 리스크가 크다. **서버(배치)에서만 평문·키 재료를 다루는 편**이 안전하다.
- React 웹 앱 프론트는 보통 **“발송 요청 / 상태 표시”**만 하고, **암호문을 넣은 HTML 파일 생성은 서버**가 담당한다.

---

## 4. 첨부 HTML을 “작은 웹앱”으로 만드는 구조

1. **뼈대 + UI**: 비밀번호 입력, 확인, 에러, 로딩. CSS는 `<style>` 인라인 권장(단일 파일).
2. **데이터 슬롯**: 서버가 **유저마다** 채운다.  
   - **옵션 A (권장·명시적)**: `<script id="payload" type="application/json">` 안에 `v`, `alg`, `iv`, `ciphertext`(Base64) 등 JSON.  
   - **옵션 B (VestMail과 유사)**: `var s = ["base64...", ...]` 처럼 **JS 변수에 Base64만** 넣는다. IV·태그는 **암호문 앞에 이진으로 붙인 뒤 한꺼번에 Base64**했을 수도 있다.
3. **복호 스크립트**: `crypto.subtle`(Web Crypto API) 등으로 **AES-GCM** 등 복호. **평문 HTML은 소스에 없음.**

서버·클라이언트가 **같은 규칙**을 써야 한다:

- 비밀번호 문자열 정규화(숫자만, 앞자리 0 등).
- **KDF**: 예) SHA-256(비밀번호 UTF-8) → 앞 16바이트를 AES-128 키로 사용 등 — **반드시 문서화·고정**.
- **알고리즘**: 예) AES-GCM + **매 메일 랜덤 IV**를 암호문과 함께 전달(또는 JSON에 `iv` 포함).

---

## 5. JSON이 없는 예시(VestMail)와의 차이

- JSON은 **필수가 아니다.** 다만 **새로 짤 때는 JSON(또는 명시적 필드)으로 `v`/`alg`/`iv`/`ciphertext`를 구분**하면 디버깅·버전 업이 쉽다.
- VestMail은 **상용·난독화**라 겉으로는 JSON이 없고 `s[]` + `eval` 뿐인 형태다. **의미는 동일: “암호문 + 복호 코드”.**

---

## 6. 보안·운영상 주의

- 첨부 HTML·JS는 **누구나 다운로드**할 수 있으므로, **오프라인 무차별 대입**에 취약할 수 있다. **6자리 숫자**는 엔트로피가 낮다.
- **로컬 파일 `file://`로 열 때** 일부 브라우저는 Web Crypto 동작이 제한될 수 있어, **“저장 후 브라우저로 열기”** 안내를 두는 경우가 많다.
- 주민번호 처리는 **개인정보보호법·내부 정책**과 맞는지 검토. 대안으로 **일회용 코드**, **앱 로그인 후만 다운로드** 등이 자주 쓰인다.

---

## 7. 새 레포에서 만들 샘플 범위 (제안)

1. **Node 스크립트**: 평문 HTML 문자열 + 비밀번호 문자열 → **AES-GCM(또는 팀이 고른 알고리즘)** 암호화 → Base64 등 출력.
2. **HTML 템플릿 1개**: 고정 UI + 고정 복호 JS + **플레이스홀더**에 암호문(JSON 또는 단일 Base64) 삽입.
3. **(선택)** `nodemailer` 등으로 **테스트 메일 발송** — 또는 파일만 생성.

배치·실제 DB 연동은 샘플 범위 밖으로 두고, **암호화 규칙과 HTML 복호가 서로 맞는지**를 최우선으로 검증하면 된다.
